Tema 01 · Fundamentos

Arquitecturas de
Base de Datos

📖 Definición
¿Qué es una arquitectura de BD?

Una arquitectura de base de datos es el conjunto de estructuras, componentes y relaciones que organizan la manera en que los datos son almacenados, gestionados y accedidos en un sistema. Su propósito fundamental es permitir la gestión eficiente, segura y confiable de los datos.

Comprender las arquitecturas es esencial para diseñar soluciones que respondan a las necesidades actuales: grandes volúmenes de datos, alta concurrencia de usuarios y estrictos requisitos de seguridad y confiabilidad.

🎯 Objetivos Principales
✦ Estandarizar el almacenamiento: definir formatos, estructuras y convenciones
✦ Optimizar el acceso a datos: minimizar tiempos de consulta y maximizar rendimiento
✦ Garantizar integridad y consistencia: asegurar que los datos representen fielmente la realidad
✦ Soportar escalabilidad: permitir el crecimiento en volumen y complejidad de datos
✦ Proporcionar seguridad: proteger la información de accesos no autorizados
Una arquitectura sólida se convierte en la columna vertebral de un sistema de información, sobre ella se construyen procesos analíticos, transaccionales y estratégicos.
🔧 Componentes Fundamentales
Toda arquitectura integra:
💾 Almacenamiento
Capa física donde residen los datos. Puede ser en disco duro, SSD, servidores distribuidos o en la nube. Su diseño determina el rendimiento y la disponibilidad.
⚙️ Motor de consultas
Software responsable de interpretar y ejecutar las instrucciones SQL. Optimiza los planes de ejecución para acelerar las búsquedas y modificaciones.
📋 Catálogo o metadata
Almacena información sobre la estructura de la BD: tablas, índices, restricciones, usuarios, vistas y permisos. Esencial para la administración del DBMS.
🔒 Capa de seguridad
Protege los datos mediante autenticación de usuarios, control de accesos, cifrado y auditorías.
🔌 Interfaces (API/ODBC/JDBC)
Permiten la comunicación entre el DBMS y las aplicaciones cliente. Aseguran la interoperabilidad entre diferentes plataformas y lenguajes de programación.

Tema 02 · Tipos

6 Tipos de Arquitecturas
de Bases de Datos

01
Centralizada
Todos los datos se almacenan en un único servidor. Facilita la administración pero limita la escalabilidad. Punto único de fallo. Ideal para instituciones educativas pequeñas.
Un servidorFácil admin.
02
Cliente-Servidor
Separa la lógica de negocio (clientes) del almacenamiento y gestión de datos (servidor). Aumenta la eficiencia y distribuye las cargas de trabajo.
Multi-usuarioSeguridad
03
Distribuida
Los datos se almacenan en múltiples nodos interconectados. Proporciona tolerancia a fallos y escalabilidad horizontal. Complejidad de administración alta.
Alta disponib.Tolerancia
04
En la Nube
Los datos se gestionan en plataformas cloud (AWS, Azure, Google Cloud). Permite elasticidad y costos ajustables. Facturación bajo demanda y gestión simplificada.
AWSAzureGCP
05
Federada
Integra múltiples bases de datos heterogéneas presentando una vista unificada. Ideal para organizaciones con sistemas legados que no pueden unificarse fácilmente.
IntegraciónVista única
06
Orientada a Servicios (SOA)
Los datos se gestionan como servicios independientes que pueden ser consumidos por diferentes aplicaciones. Muy utilizada en microservicios y arquitecturas modernas.
Microservs.APIs
⚖️ Relación con Requisitos del Sistema
Factores para elegir la arquitectura
✦ Rendimiento: distribuidas y cliente-servidor permiten paralelismo y mayor velocidad
✦ Disponibilidad: nube y distribuidas garantizan alta disponibilidad mediante replicación y balanceo
✦ Consistencia: centralizadas y cliente-servidor facilitan la consistencia fuerte
✦ Escalabilidad: nube y distribuidas se ajustan dinámicamente a la demanda
💡 Ejemplos por sector
Casos de uso reales
🏦 Banco: arquitectura cliente-servidor para separar operaciones de usuarios y procesamiento de datos
📱 Red social: sistema distribuido para manejar millones de interacciones
🚀 Startup: aplicación en la nube para escalar según la demanda
🏛️ Universidad: cliente-servidor para gestión académica multiusuario

Tema 03 · Modelado

Modelado de Datos
y Niveles de Abstracción

📐 ¿Qué es el Modelado de Datos?

El modelado de datos es un proceso clave en el diseño de bases de datos que busca representar, de manera estructurada, cómo se organizan, relacionan y almacenan los datos dentro de un sistema de información. Permite transformar los requisitos del negocio en un diseño técnico implementable en un gestor de bases de datos (DBMS).

Incluye la normalización para garantizar la integridad y consistencia, así como la desnormalización y materialización cuando es necesario optimizar el rendimiento.

🛠️ Técnicas de Modelado
Modelo ER (Entidad-Relación)
Representa entidades, atributos y relaciones. Ampliamente usado en bases de datos transaccionales. Herramienta: diagrama Entidad-Relación.
UML (Unified Modeling Language)
Utiliza diagramas de clases para representar datos en el contexto de sistemas orientados a objetos. Ideal para sistemas bancarios.
Modelos Dimensionales
Esquema estrella (star schema) y copos de nieve (snowflake) para entornos de Business Intelligence (BI). Optimizan consultas analíticas sobre grandes volúmenes de datos.
📊 Niveles de Abstracción
Los 3 niveles del proceso de modelado
🔵 Nivel Conceptual
Representa los datos a un nivel general, sin atender aspectos técnicos. Su objetivo es reflejar las entidades del negocio, sus atributos y las relaciones entre ellas. Herramienta típica: diagrama Entidad-Relación (ER).
🟡 Nivel Lógico
Traduce el modelo conceptual a estructuras comprensibles por un DBMS: tablas, claves primarias y foráneas, relaciones y restricciones. Se centra en la organización sin considerar aún la implementación física.
🟢 Nivel Físico
Corresponde a la implementación en el DBMS elegido. Define estructuras físicas de almacenamiento, índices, particiones, tipos de datos y consideraciones de rendimiento.
Trabajar en niveles asegura la trazabilidad entre lo que necesita la organización y lo que se implementa en el sistema.

Tema 04 · Normalización

Normalización, Formas Normales
y Desnormalización

📏 Normalización
¿Qué es?

La normalización es el proceso de organizar las tablas y relaciones de una base de datos para reducir la redundancia y mejorar la integridad de los datos. Las formas normales más comunes son:

1FN — Primera Forma Normal
Elimina grupos repetidos y asegura atomicidad. Cada celda contiene un único valor.
2FN — Segunda Forma Normal
Elimina dependencias parciales de claves compuestas. Cada atributo depende de toda la clave primaria.
3FN — Tercera Forma Normal
Elimina dependencias transitivas. Los atributos no clave dependen únicamente de la clave primaria.
BCNF — Forma Normal de Boyce-Codd
Forma avanzada que refuerza la 3FN. Toda determinante es una superclave.
📈 Impacto en la Arquitectura
✦ Mejora la consistencia e integridad de datos
✦ Reduce redundancia, aunque puede generar mayor complejidad en consultas
✦ Obliga a establecer relaciones más precisas entre tablas
🔄 Desnormalización y Materialización
Cuándo y por qué aplicarlas

La desnormalización es el proceso inverso: combinar tablas o duplicar datos con el fin de mejorar el rendimiento en consultas específicas.

La materialización consiste en crear vistas materializadas o tablas que almacenan resultados de consultas frecuentes para acelerar el acceso a la información.

Cuándo aplicarlas:
✦ En sistemas de análisis (BI) donde prima la velocidad de respuesta
✦ Cuando las consultas son más frecuentes que las actualizaciones
✦ Cuando los costos de acceso a datos distribuidos superan los beneficios de la normalización
Son esenciales en entornos de Big Data y Data Warehousing. Reducen la latencia en consultas complejas y mejoran la experiencia de usuario.
💼 Ejemplos Prácticos
🏫 Normalización: Un colegio normaliza su BD de estudiantes para evitar duplicación de nombres y cursos → 3FN
🛍️ Desnormalización: Una cadena de supermercados desnormaliza datos de inventarios para consultas rápidas de ventas
🎓 Materialización: Una universidad crea vistas materializadas para consultas frecuentes de calificaciones, evitando recalcular datos en cada acceso
📡 Telecomunicaciones: Una empresa implementa desnormalización en su data warehouse para reportes diarios de tráfico de red

📊 Evidencia 1 · Diagrama de Arquitectura
Diagrama de Arquitectura de Base de Datos

Diagrama elaborado en Draw.io / Lucidchart / MySQL Workbench que representa la arquitectura elegida para un sistema real.

Sistema hospitalario: ¿Qué arquitectura elegir y por qué?
Se recomienda una arquitectura cliente-servidor con extensión distribuida. El servidor central gestiona los datos clínicos de pacientes (historiales, diagnósticos, medicamentos), mientras que múltiples clientes acceden desde consultorios, urgencias y laboratorios. Los módulos críticos como UCI se replican en nodos distribuidos para garantizar disponibilidad continua. Componentes del diagrama: Capa de presentación (aplicaciones web/escritorio), capa lógica (servidor de aplicaciones), capa de datos (BD central + réplicas), capa de seguridad (autenticación LDAP + cifrado TLS).
Sistema universitario: diseño de arquitectura priorizando integridad de calificaciones
Arquitectura cliente-servidor con BD centralizada. La consistencia fuerte es prioritaria para las notas. Arquitectura: servidor PostgreSQL centralizado con transacciones ACID, clientes web para docentes y estudiantes, módulo de reportes con vistas materializadas para consultas frecuentes, y backup automatizado diario. La centralización garantiza que todos los usuarios vean los mismos datos en tiempo real, evitando discrepancias en registros académicos.
📄 Evidencia 2 · Documento de Diseño Lógico
Diseño Lógico con Modelos ER/UML y Normalización

Documento que incluye definición de arquitectura, componentes explicados, justificación por requisitos y diagramas según el caso.

Diseño lógico para sistema académico: modelo ER → lógico → físico
Nivel Conceptual (ER): Entidades: Estudiante, Curso, Docente, Matrícula, Nota. Relaciones: Estudiante se matricula en Curso, Docente imparte Curso, Matrícula genera Nota.

Nivel Lógico: Tablas: ESTUDIANTE(id, nombre, apellido, dni), CURSO(id, nombre, creditos, ciclo), MATRICULA(id_est, id_curso, fecha, estado), NOTA(id_mat, parcial1, parcial2, final). Clave primaria compuesta en MATRICULA.

Nivel Físico (SQL Server/PostgreSQL): Índices en id_estudiante y id_curso para optimizar búsquedas. Tipos de datos: VARCHAR(100) para nombres, DECIMAL(4,2) para notas, DATE para fechas. Particionamiento por ciclo académico.
Aplicar normalización hasta 3FN en tabla de pedidos con redundancias
Tabla original (sin normalizar): Pedido(id_pedido, cliente_nombre, cliente_dir, producto, precio, categoria).
1FN: Separar valores atómicos → no hay grupos repetidos.
2FN: Separar CLIENTE(id_cliente, nombre, dirección) y PRODUCTO(id_producto, nombre, precio, categoria).
3FN: Si categoría depende solo del tipo de producto → CATEGORIA(id_cat, nombre), PRODUCTO referencia id_cat. Resultado: 4 tablas normalizadas (PEDIDO, CLIENTE, PRODUCTO, CATEGORIA) con relaciones mediante claves foráneas.
🏗️ Evidencia 3 · Sistema Real
Representación de sistema real en diagrama
Propuesta de desnormalización para mejorar rendimiento de reportes
Para un sistema de ventas con reportes diarios por región y producto, se aplica desnormalización creando una tabla REPORTE_VENTAS_DIARIAS(fecha, región, producto, cantidad_total, monto_total) que pre-agrega los datos. Esto reduce el tiempo de consulta de reportes de 15 segundos a menos de 1 segundo, eliminando la necesidad de joins complejos entre 5 tablas normalizadas. Se actualiza mediante un proceso batch nocturno.
